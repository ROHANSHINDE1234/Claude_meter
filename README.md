# Claude Meter — Project Status & Handoff

**Project:** ESP32 + OLED desk display showing live Claude Code usage
**Inspired by:** [Clawdmeter](https://github.com/HermannBjorgvin/Clawdmeter) — adapted to run on different, simpler hardware
**Purpose of this document:** Handoff reference so a new conversation (or you, later) can pick this project up without re-explaining everything from scratch.

---

## 1. Project Context (Read First)

This is a simplified, from-scratch reimplementation of the idea behind Clawdmeter — a small desk display that shows Claude Code usage — built around hardware already on hand rather than the original project's touchscreen AMOLED board. The original Clawdmeter uses BLE + LVGL + a touchscreen; **none of that applies here.** This project uses a plain I2C OLED, no touchscreen, no BLE, and (as of the current decision) no WiFi either — see section 3.

### 1.1 Hardware in use

| Component | Identification | Notes |
|---|---|---|
| Microcontroller | Generic ESP32-WROOM-32 DevKit, 38-pin | CP2102 USB-to-UART chip on board; sold under many names (DOIT DevKit V1, "ESP32 Dev Module," various AliExpress clones) — all the same core module |
| Display | 1.3" I2C OLED, **SH1106 driver**, 128×64 | Board is printed **"1.30'' IIC V2.2"** — this confirms SH1106, *not* SSD1306, despite looking visually identical to the more common 0.96" SSD1306 modules |
| Button | Generic momentary push button | Wired to GPIO27 + GND, using internal pull-up (no external resistor needed) |

### 1.2 Wiring (confirmed working)

| ESP32 pin | Connects to | Purpose |
|---|---|---|
| 3V3 | OLED VCC | Power |
| GND | OLED GND | Ground |
| GPIO22 | OLED SCK (SCL) | I2C clock |
| GPIO21 | OLED SDA | I2C data |
| GPIO27 | Button leg 1 | Digital input, `INPUT_PULLUP` |
| GND | Button leg 2 | Ground |

## 2. Toolchain (Deliberate, Non-Negotiable Choice)

Following the same philosophy used on the separate STM32 firmware project: **no vendor-specific IDE.** VS Code is used purely as a text editor; all compiling and flashing happens through a CLI tool.

- **arduino-cli** (not the Arduino IDE GUI, not PlatformIO — same reasoning as rejecting PlatformIO on the STM32 side: avoid extra build-system abstraction)
- Installed version: **1.5.1**
- Installed manually — not available via `winget search arduino-cli`, so the binary was downloaded directly from `github.com/arduino/arduino-cli/releases` and its folder added to PATH
- ESP32 core installed: **esp32:esp32@3.3.10**
- Board identifier (FQBN): **`esp32:esp32:esp32`** (maps to "ESP32 Dev Module")
- Libraries installed: `Adafruit SH110X`, `Adafruit GFX Library`, `Adafruit BusIO` (auto-installed dependency)
  - Note: `Adafruit SSD1306` was also installed early on but is **not used** — wrong driver for this display, see section 4's known issues

### 2.1 Where things live

- Toolchain, cores, libraries → `%LOCALAPPDATA%\Arduino15\` (default location, intentionally *not* vendored into the project folder — shared across any future sketch, same reasoning as not vendoring the ARM toolchain into the STM32 repo)
- Actual project code → `Documents\Claude_meter\` (this is the folder open in VS Code)

### 2.2 Project folder structure (current)

```
Claude_meter/

  oled_test/
```

## 3. Key Architecture Decisions

### 3.1 Connectivity: USB Serial (decided after exploring alternatives)

The original plan was WiFi, connecting to a mobile hotspot since the office has no WiFi. Several transport options were evaluated before settling on the final approach:

| Option | Why it was set aside |
|---|---|
| WiFi + mobile hotspot, HTTP polling | Phone hotspots don't allow a static IP for the laptop — the ESP32 would need to rediscover the daemon's address every session |
| MQTT via a broker | Solves the IP problem (both sides connect outward to a shared broker) but adds a broker dependency for no longer benefit |
| BLE (the original Clawdmeter's approach) | Avoids network/IP issues entirely, but was a bigger departure from where the project had already gone |
| **USB Serial** ✅ **chosen** | Once confirmed the ESP32 will always sit right next to the laptop and a cable is acceptable, this became the obvious simplest choice — no WiFi credentials, no IP discovery, no broker, and the same cable already powers the board |

**Implication:** the ESP32 itself never touches the internet. Only the laptop-side daemon needs internet access (to call the Anthropic API) — matches the original Clawdmeter's philosophy of keeping the OAuth token off the microcontroller entirely.

### 3.2 Button's role changed

Originally planned as a "force WiFi reconnect" button. Since WiFi was dropped in favor of Serial, it's been repurposed as a **manual force-refresh** trigger — pressing it will (once implemented) send a `REFRESH` line over serial to ask the daemon to poll Claude usage immediately, rather than waiting for the next scheduled interval.

### 3.3 Display driver: SH1106, not SSD1306

Both are extremely common on visually identical 0.96"/1.3" blue OLED boards, but they are **not** software-compatible. This board is confirmed SH1106 (see section 1.1) and must use the `Adafruit_SH110X` library and `Adafruit_SH1106G` class — any future display code must keep using this, not `Adafruit_SSD1306`.

## 5. Known Issues and Fixes

| Symptom | Cause | Fix |
|---|---|---|
| `'arduino-cli' is not recognized` | Not installed / not on PATH | Downloaded release zip manually from GitHub, extracted `arduino-cli.exe`, added its folder to PATH, opened a **new** terminal window |
| `winget search arduino-cli` found nothing | Not published to winget | Manual download instead |
| `arduino-cli board list` → "No boards found" | CP2102 chip showed in Device Manager under **Other devices** with a yellow warning triangle — driver missing | Downloaded Silicon Labs' official CP210x Universal Windows Driver, used **Device Manager → Update driver → Browse my computer → point at extracted `x64` folder** (the driver's own `.exe`/`.bat` files in the package are not a one-click installer) |
| Compile appeared to hang with no output | Normal — first-time ESP32 compile is slow (compiling a large core from scratch); pressing Ctrl+C interrupts a build that was actually still working | Rerun and just wait; a minute or more of silence on first compile is expected |
| OLED showed scattered static instead of text | Code used `Adafruit_SSD1306`, but the board is SH1106-driven | Installed `Adafruit SH110X`, switched to `Adafruit_SH1106G` class and `SH110X_WHITE` constant |
| VS Code Problems panel shows red "cannot open source file" errors | IntelliSense (VS Code's C/C++ extension) uses a separate include-path system from the real `arduino-cli` compiler — doesn't affect actual builds | In progress: generating `compile_commands.json` via `--only-compilation-database` and pointing `c_cpp_properties.json` at it, since manually listing every ESP-IDF component path (there are dozens bundled into the ESP32 3.x core) isn't practical |

## 6. Quick Reference Commands

| Task | Command |
|---|---|
| Check port | `arduino-cli board list` |
| Compile | `arduino-cli compile --fqbn esp32:esp32:esp32 oled_test` |
| Upload | `arduino-cli upload -p COM3 --fqbn esp32:esp32:esp32 oled_test` |
| Regenerate IntelliSense data | `arduino-cli compile --fqbn esp32:esp32:esp32 --build-path build --only-compilation-database .` (run from inside `oled_test/`) |
| Install a new library | `arduino-cli lib install "Library Name"` |
