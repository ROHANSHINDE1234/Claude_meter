# Claude Meter — Project Status & Handoff

**Project:** ESP32 + OLED desk display showing live Claude Code usage
**Inspired by:** [Clawdmeter](https://github.com/HermannBjorgvin/Clawdmeter) — adapted to run on different, simpler hardware
**Purpose of this document:** Handoff reference so a new conversation (or you, later) can pick this project up without re-explaining everything from scratch. See also [CLAUDE.md](CLAUDE.md) — the condensed operating rules Claude Code loads automatically in this repo (commands, non-negotiable architecture decisions); this README remains the full narrative history and reference.

## 1. Project Context (Read First)

This is a simplified, from-scratch reimplementation of the idea behind Clawdmeter — a small desk display that shows Claude Code usage — built around hardware already on hand rather than the original project's touchscreen AMOLED board. The original Clawdmeter uses BLE + LVGL + a touchscreen; **none of that applies here.** This project uses a plain I2C OLED and no touchscreen, no BLE, no LVGL. It **does** use WiFi — see section 3.1 for why that's a reversal of the project's earlier USB-Serial-only decision.

### 1.1 Hardware in use

| Component | Identification | Notes |
|---|---|---|
| Microcontroller | Generic ESP32-WROOM-32 DevKit, 38-pin | CP2102 USB-to-UART chip on board; sold under many names (DOIT DevKit V1, "ESP32 Dev Module," various AliExpress clones) — all the same core module |
| Display | 1.3" I2C OLED, **SH1106 driver**, 128×64 | Board is printed **"1.30'' IIC V2.2"** — this confirms SH1106, *not* SSD1306, despite looking visually identical to the more common 0.96" SSD1306 modules |
| Button | Generic momentary push button | Wired to GPIO27 + GND, using internal pull-up (no external resistor needed) |

### 1.2 Wiring (confirmed working)

| ESP32 pin | Connects to | Purpose |
|---|---|---|
| 3V3 | OLED VCC | Power |
| GND | OLED GND | Ground |
| GPIO22 | OLED SCK (SCL) | I2C clock |
| GPIO21 | OLED SDA | I2C data |
| GPIO27 | Button leg 1 | Digital input, `INPUT_PULLUP` |
| GND | Button leg 2 | Ground |

## 2. Toolchain (Deliberate, Non-Negotiable Choice)

Following the same philosophy used on the separate STM32 firmware project: **no vendor-specific IDE.** VS Code is used purely as a text editor; all compiling and flashing happens through a CLI tool.

- **arduino-cli** (not the Arduino IDE GUI, not PlatformIO — same reasoning as rejecting PlatformIO on the STM32 side: avoid extra build-system abstraction)
- Installed version: **1.5.1**
- Installed manually — not available via `winget search arduino-cli`, so the binary was downloaded directly from `github.com/arduino/arduino-cli/releases` and its folder added to PATH
- ESP32 core installed: **esp32:esp32@3.3.10**
- Board identifier (FQBN): **`esp32:esp32:esp32`** (maps to "ESP32 Dev Module")
- Libraries installed: `Adafruit SH110X`, `Adafruit GFX Library`, `Adafruit BusIO` (auto-installed dependency)
  - Note: `Adafruit SSD1306` was also installed early on but is **not used** — wrong driver for this display, see section 4's known issues

### 2.1 Where things live

- Toolchain, cores, libraries → `%LOCALAPPDATA%\Arduino15\` (default location, intentionally *not* vendored into the project folder — shared across any future sketch, same reasoning as not vendoring the ARM toolchain into the STM32 repo)
- Actual project code → `Documents\Claude_meter\` (this is the folder open in VS Code)

## 3. Key Architecture Decisions

### 3.1 Connectivity: direct WiFi + HTTPS (supersedes the earlier USB-Serial decision)

The original plan — and what's described further down in this section's history — was WiFi, then a mobile-hotspot IP problem pushed the project to USB Serial with a laptop-side daemon relaying to the Anthropic API. That reasoning is preserved below for context, but **it no longer reflects the current design.**

**What changed:** the device is now used at home, on a WiFi network the ESP32 can join directly and stay on — the original mobile-hotspot static-IP problem that ruled out WiFi doesn't apply in this environment. `api.anthropic.com` is a fixed public hostname (unlike a laptop-side daemon on the local network), so the ESP32 can just:

```
ESP32 → home WiFi → router → internet → api.anthropic.com
```

No laptop, no daemon, no relay, no IP discovery. Full rationale, the data-source investigation (there's no dedicated usage endpoint — usage is read off response headers of a real, tiny inference call), and the credential-handling trade-offs are in [token-usage-feature-decision.md](token-usage-feature-decision.md).

**Original transport evaluation (superseded, kept for history):**

| Option | Why it was set aside |
|---|---|
| WiFi + mobile hotspot, HTTP polling | Phone hotspots don't allow a static IP for the laptop — the ESP32 would need to rediscover the daemon's address every session |
| MQTT via a broker | Solves the IP problem (both sides connect outward to a shared broker) but adds a broker dependency for no longer benefit |
| BLE (the original Clawdmeter's approach) | Avoids network/IP issues entirely, but was a bigger departure from where the project had already gone |
| USB Serial (chosen at the time, now superseded) | Assumed the ESP32 would always sit right next to the laptop; once the device moved to a home-WiFi environment, direct WiFi became viable and simpler than maintaining a laptop-side daemon |

**Implication of the new design:** the ESP32 itself now touches the internet and holds the Claude OAuth access token in flash — a real change in risk profile from the original "keep the token off the microcontroller entirely" philosophy. See section 6 of the decision doc ("Open Decisions") for the trade-offs; the mitigation adopted for now is Approach A (no on-device token refresh — see section 3.2 below) plus keeping the token out of git via `claude_meter/include/secrets.h`.

### 3.2 Button's role

Originally planned as a "force WiFi reconnect" button, then briefly a "send `REFRESH` over serial" button during the USB-Serial design. With the ESP32 now calling Anthropic directly (section 3.1), it's simpler still: pressing it calls `TokenUsageManager::forceRefresh()` directly in `claude_meter.ino`, polling Claude usage immediately instead of waiting for the next scheduled interval.

### 3.3 Display driver: SH1106, not SSD1306

Both are extremely common on visually identical 0.96"/1.3" blue OLED boards, but they are **not** software-compatible. This board is confirmed SH1106 (see section 1.1) and must use the `Adafruit_SH110X` library and `Adafruit_SH1106G` class — any future display code must keep using this, not `Adafruit_SSD1306`.

### 3.4 Updating the access token without a reflash: local web UI, not auto-refresh or a relay

Once the device moves from home WiFi to an office mobile hotspot, `secrets.h` + USB reflash every time the token expires stops being workable — the ESP32 won't be sitting next to a laptop. Three alternatives were considered:

| Option | Why it was set aside / adopted |
|---|---|
| ESP32 auto-refreshes its own OAuth token | Rejected outright, even before this discussion — see section 3.1's implication and `token-usage-feature-decision.md` §4: it can race the Claude Code CLI's own refresh and lock you out of `claude login` on your actual machine. |
| Console API key instead of OAuth session token | Rejected — solves the expiry problem but changes what's displayed. An API key authenticates via `x-api-key`, not the OAuth session, so the rate-limit headers it returns reflect the **API tier's** request/token-per-minute usage, not "% of your Pro/Max 5-hour window." For light polling traffic that number sits near 0% — not the metric this project exists to show. Also requires loading paid credits. |
| A relay (laptop or small always-on device pushes a fresh token to a cloud store; ESP32 polls it) | Solves the problem completely, but needs *something* to stay logged into Claude Code and online continuously to keep re-minting the token. No always-on device available (a Raspberry Pi or similar would work, but that's new hardware/infra, not adopted for now). |
| **Local web UI on the ESP32** ✅ **chosen** | The token now lives in flash (NVS, via `TokenStore`) instead of only the `secrets.h` macro. `TokenWebServer` serves a small password-gated page at `http://<esp32-ip>/` where a fresh token can be pasted in ~10 seconds — no USB, no recompile, no reflash. |

**Finding the page — QR code, not typing the IP.** After WiFi connects, the OLED shows a QR code encoding `http://<esp32-ip>/` (rendered by `QRCodeScreen`, using ESP-IDF's bundled `qrcode` component — see the note below on why that specific library) — scan it with a phone camera to open the page directly, no typing a dotted-quad IP by hand. This screen isn't on a fixed timer: it stays up until the **first successful usage fetch** (`claude_meter.ino`'s `waitingForFirstFetch` flag, gated on `usageManager.current().valid` — once true, `displayManager.setScreen(&usageScreen)` swaps it out), so if the stored token is bad or expired, the QR just keeps showing — which is exactly when you need to scan it.

**Why ESP-IDF's `qrcode` component and not an installed library:** the ESP32 core bundles its own QR code generator (`espressif__qrcode`, header `qrcode.h`) as part of the platform's system includes. A separately-installed Arduino library with the same header name (`QRCode` by ricmoo) was tried first and silently lost the header collision — the ESP-IDF one wins regardless of what's in `Documents/Arduino/libraries/`. Once that was understood, the ESP-IDF component turned out to be the better fit anyway: it auto-selects the smallest QR version that fits the given text, instead of requiring a hardcoded version guess.

**Submitting a token now triggers an immediate check, not a silent save.** `TokenWebServer::handleUpdate()` calls `TokenUsageManager::forceRefresh()` right after saving the new token to `TokenStore`, and the page's response tells you whether that fetch actually succeeded ("Token updated and verified" vs. "Token saved, but the verification fetch failed — double-check the value"). Without this, the QR would sit on screen for up to `POLL_INTERVAL_MS` (5 minutes) after a successful submit before the usage screen appeared — the button would still force it sooner, but this closes the gap without needing to reach for the button at all.

**How the workflow actually looks day to day:** the office PC already runs Claude Code under the same account, so `%USERPROFILE%\.claude\.credentials.json` there has its own actively-refreshing token — no admin rights needed, it's just a file. The remaining manual step is getting that token text onto whatever device can reach the ESP32's page: if the ESP32 is on a phone's hotspot, the phone is on the same local network as the ESP32 (the office PC, on the office's own WiFi, is not) — so copy the token from the office PC to the phone (clipboard sync, or a private note to yourself) and paste it into the page from there. **Note:** `WIFI_SSID`/`WIFI_PASSWORD` are still compiled-in constants from `secrets.h` — the web UI only updates the *token*, not WiFi credentials. If the office hotspot's name/password ever differs from what's currently flashed, that still needs a `secrets.h` edit + reflash.

**Security note:** `TokenWebServer`'s password check is a shared-secret deterrent for a home/hotspot-only device, not real authentication — plain HTTP, no TLS, no rate limiting. Don't reuse a password that matters, and don't expose this port beyond a trusted local network.

### 3.5 Code layout: `include/` + `src/`, and a real Arduino build-system gotcha

The `claude_meter/` sketch splits headers into `include/` and implementation into `src/`, with `claude_meter.ino` reduced to pure orchestration (construct every module, wire them together, hand off between the boot QR screen and the usage screen). This wasn't arbitrary — it came from actually testing how `arduino-cli` compiles a sketch, because the rules are stricter than they look:

- **Only a folder literally named `src/` gets auto-compiled.** Any other subfolder name (tested: an arbitrary `plainsubdir/`) is silently skipped — its `.cpp` files never even reach the linker, no warning given. This is why implementation files live in `src/` specifically, not e.g. `impl/` or `modules/`.
- **Nested subfolders inside `src/` do work** (confirmed: `src/network/Bar.cpp` compiles and links fine) — but only if referenced correctly, see next point.
- **Bare `#include "Foo.h"` only resolves within the same folder.** There's no single flat include path across a sketch's subfolders the way library `src/` folders sometimes behave. Concretely: `claude_meter.ino` (in the sketch root) must write `#include "include/Foo.h"` for every project header; every `.cpp` in `src/` writes `#include "../include/Foo.h"` for its own header and any others it needs; headers inside `include/` can freely bare-include each other (`#include "IScreen.h"`) since they're siblings there.

Two smaller modules exist specifically because of an earlier restructure pass, and are worth knowing about before assuming `claude_meter.ino` or `DisplayManager` is the right place to add something:

- **`WiFiConnection`** owns `WiFi.begin`/the blocking connect loop/`configTime()` NTP setup — pulled out of `claude_meter.ino` so the .ino stays orchestration-only. `ClaudeSessionProvider` still owns the actual HTTPS call and just checks `WiFi.status()` before using it.
- **`QRCodeScreen`** implements `IScreen` (same interface as `TokenUsageScreen`) instead of being a special "bypass the active screen" method on `DisplayManager`. `DisplayManager` itself is back to exactly what it says on the tin: own the OLED, own whichever `IScreen*` is active, nothing else. `setScreen()` renders immediately (not on the next periodic tick) so screen swaps — boot QR → usage screen — still appear instantly.
